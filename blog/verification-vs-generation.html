A Clean Test for Whether Verification Is Easier Than Generation

May 24, 2026

I want to measure two things about the same model. First: if I give it only a problem, how often does it solve the problem correctly? Second: if I give it a problem and a candidate answer, how often can it tell whether that answer is correct?

The tempting move is to report two numbers: generation accuracy and verification accuracy. But that comparison is too loose. Verification accuracy is not an intrinsic property of the model. It depends on where the candidate answers came from, how subtle the wrong answers are, and whether the model is being asked to judge its own outputs.

Here is the cleaner object: measure generation and verification at the problem level, use a different but capability-matched model to generate candidate answers, and then ask where verification remains easier than generation as candidate-answer subtlety increases.

Definitions

Let $D$ be a distribution over problems $x$, and let $M$ be the target model we want to study. Let $C(x, y) \in \{0, 1\}$ denote whether answer $y$ is correct for problem $x$.

The generation accuracy of $M$ is:

$$G_M = P_{x \sim D}\left[C(x, M(x)) = 1\right].$$

For verification, introduce a separate answer-generating model $A$. The verifier $M$ receives $(x, y_A)$ and predicts whether $y_A$ is correct. The first important point is that verification should be split into two rates:

$$V^+_{M,A} = P[M \text{ says correct} \mid C(x, y_A)=1],$$

$$V^-_{M,A} = P[M \text{ says wrong} \mid C(x, y_A)=0].$$

$V^+$ is the true positive rate: does the model accept correct answers? $V^-$ is the true negative rate: does the model reject wrong answers?

The raw verification accuracy is usually the wrong summary statistic, because it depends on the base rate of correct candidate answers produced by $A$:

$$V^{raw}_{M,A} = q_A V^+_{M,A} + (1-q_A)V^-_{M,A},$$

where $q_A = P[C(x, y_A)=1]$. A stronger answer generator changes $q_A$, so raw accuracy can move even if the verifier did not get better. The more stable number is balanced verification accuracy:

$$B_{M,A} = \frac{V^+_{M,A} + V^-_{M,A}}{2}.$$

So the basic comparison should not be $G_M$ versus one generic $V_M$. It should be $G_M$ versus $B_{M,A}$ for a specified answer source $A$.

Avoid Self-Endorsement

If $M$ generates an answer and then $M$ verifies that same answer, the experiment mixes two effects: verification ability and self-endorsement. The model may fail to reject its own wrong answer for reasons that have little to do with whether verification is generally easier than generation.

So the primary experiment should use $A \ne M$. Ideally, $A$ should be close to $M$ in capability but from a different model family. If $A$ is much weaker, its errors are too obvious and verification accuracy is inflated. If $A$ is much stronger, it may produce too few wrong answers and the remaining errors may be unusually subtle.

The cleanest design is not to pick only one $A$, but to sweep over a ladder of answer generators:

$$A_1, A_2, \ldots, A_k,$$

from weak to capability-matched to stronger. This gives a curve:

$$A_i \mapsto B_{M,A_i}.$$

That curve is more informative than a yes-or-no answer. It tells us what level of error subtlety the target model can still verify.

The 2x2 Table

To connect generation and verification, compute both outcomes per problem. For each problem $x$, define:

  • $S_x = 1$ if $M$ solves $x$ correctly when given only the problem.
  • $Z_x = 1$ if $M$ verifies the candidate set for $x$ correctly.

For $Z_x$, a good candidate set contains at least one correct answer and one wrong answer from $A$ or from a controlled perturbation. This prevents the verifier from looking good by exploiting the base rate of correctness.

Fails generation $p_{01}$ $p_{00}$

Now the two original numbers are no longer floating separately:

$$G_M = p_{11} + p_{10},$$

$$P[Z_x=1] = p_{11} + p_{01}.$$

The most important cell is $p_{01}$: problems the model cannot solve, but can verify. This is the direct empirical signature of prover-verifier asymmetry. If verification is genuinely easier than generation, this cell should be nontrivially large, especially in the capability-matched regime.

The other interesting cell is $p_{10}$: problems the model can solve, but cannot verify. This says generation and verification are not the same capability run in reverse. The model may arrive at an answer through pattern completion or fragile heuristics without having a robust criterion for recognizing correctness.

Useful Derived Quantities

The first derived quantity is the verification advantage ratio:

$$VAR_{M,A} = P[Z_x=1 \mid S_x=0] = \frac{p_{01}}{p_{01}+p_{00}}.$$

This asks: among problems $M$ cannot solve, how often can it still verify the answer source? This is the cleanest number for the asymmetry story. A large $VAR$ means the model has latent discriminative knowledge that it cannot reliably convert into generation.

The second quantity is the verification advantage over generation:

$$\Delta_{M,A} = B_{M,A} - G_M.$$

$\Delta_{M,A} > 0$ says verification beats generation against answer source $A$. But $\Delta$ should always be reported with $A$ and with the correctness base rate $q_A$, because changing the answer source changes the task.

The third quantity is the self-endorsement gap. This should be a secondary diagnostic, not the primary comparison:

$$SEG_M = V^-_{M,A} - V^-_{M,self}.$$

If $SEG_M$ is large, the model is substantially worse at rejecting its own wrong answers than at rejecting another model's wrong answers. That does not refute prover-verifier asymmetry; it says self-verification is contaminated by self-endorsement.

Experimental Protocol

  1. Choose a benchmark with reliable ground truth and stratify problems by difficulty.
  2. Choose a target model $M$.
  3. Choose auxiliary generators $A_1,\ldots,A_k$, preferably from different model families and spanning weak to capability-matched to stronger.
  4. Measure $G_M$ by asking $M$ to solve each problem directly.
  5. For each $A_i$, generate candidate answers $y_{A_i}$ for the same problems.
  6. Label each candidate answer as correct or wrong using the benchmark answer, a trusted grader, or human adjudication.
  7. Ask $M$ to verify each $(x, y_{A_i})$ pair with a fixed binary prompt.
  8. Compute $V^+$, $V^-$, $B$, the problem-level 2x2 table, $VAR$, and $\Delta$ for each $A_i$ and each difficulty bucket.
  9. Bootstrap confidence intervals over problems, and treat problem ID as the unit of resampling.

The core plots are simple:

  • $B_{M,A_i}$ versus $G_{A_i}$, using $G_{A_i}$ as a proxy for answer subtlety.
  • $VAR_{M,A_i}$ versus $G_{A_i}$.
  • $p_{01}$ by problem difficulty bucket.
  • $V^-_{M,A_i}$ versus $V^-_{M,self}$, if self-endorsement is part of the story.

How to Read the Result

If $B_{M,A}$ is higher than $G_M$ only when $A$ is weak, then verification is catching obvious mistakes, not demonstrating a deep generation-verification gap.

If $B_{M,A}$ remains higher than $G_M$ for a capability-matched $A$, and $VAR_{M,A}$ is high, then we have strong evidence that the model can recognize correct solutions in cases where it cannot produce them.

If $p_{01}$ is large in medium-difficulty buckets but collapses on the hardest bucket, then verification is easier only in an intermediate regime. That is still useful: it identifies where model-assisted eval generation can work and where it needs stronger tools, humans, or external checkers.

If $V^-_{M,self}$ is much lower than $V^-_{M,A}$, then the blocker for self-improvement is not verification in general. The blocker is self-endorsement. In that case, the next experiment is to test whether lightweight transformations of the answer, the verifier prompt, or the model's internal representation can raise $V^-_{M,self}$ without reducing $V^+$.

The useful conclusion is not a single scalar statement like "verification is easier than generation." The useful conclusion is a boundary: verification is easier than generation for this target model, against this level of answer subtlety, in this difficulty regime, with this much self-endorsement removed.

That boundary is exactly what a self-evolving eval system needs to know. It tells us when a model can help generate and filter new evaluation cases, and when the eval loop has to bring in another model, a stronger verifier, or a non-model ground-truth mechanism.
